# Athena Core v1 Specification

Version: 1.0

Status: Frozen

---

# Purpose

Athena Core provides the minimum cognitive architecture required for a language model to function as a persistent personal assistant.

The Core is intentionally small.

Its purpose is to provide stable cognitive foundations upon which future capabilities and extensions can be built.

The Core should change rarely.

---

# Design Philosophy

Athena augments a language model rather than replacing it.

The language model performs reasoning.

Athena provides persistent cognition through memory, learning, orchestration, and future extensibility.

Every component in the Core has a single responsibility.

---

# Core Components

## Brain

Responsibilities:

- Coordinate the cognitive pipeline.
- Create and manage Thoughts.
- Own shared managers and providers.
- Orchestrate cognitive workers.

The Brain does not perform reasoning.

---

## Thought

A Thought is Athena's temporary cognitive workspace for processing one interaction.

A Thought exists only during a single cognitive cycle.

It contains:

- user input
- conversation history (Working Memory)
- retrieved memories (Episodic)
- retrieved knowledge (Semantic)
- planner_decision (Tool Planner output)
- tool_context (Tool Router output)
- reasoning_package (Context Budget Manager output for PromptBuilder)
- learning_package (Context Budget Manager output for KnowledgeExtractor)
- metadata
- trace information
- response
- candidates (Knowledge Extractor output)
- reflection

A Thought is destroyed after processing completes.

---

## Memory Manager

The Memory Manager provides access to Athena's memory systems.

Responsibilities:

- Episodic Memory
- Semantic Memory
- Temporary learning state

---

## Semantic Memory

Semantic Memory stores durable facts.

It is the only long-term memory consulted during reasoning.

Semantic Memory is updated only through the Learning Pipeline.

---

## Tool Planner

Responsibilities:

- Decide whether an external tool is needed for the current user input.

Inputs:

- User input
- Conversation history
- Semantic Memory

Outputs:

- PlannerDecision (tool to invoke, query, prompt, reason)

The Tool Planner does NOT execute tools. It only produces a decision.

---

## Tool Router

Responsibilities:

- Execute the requested tool and produce ToolContext.

Inputs:

- PlannerDecision

Outputs:

- ToolContext (tool_name, content, priority, learning_visible)

The Tool Router is the ONLY component that invokes tool implementations.

Supported tools: /system (System Snapshot), /web (Web Search via DuckDuckGo).

---

## ToolContext

Temporary context generated by an Athena native tool.

Metadata fields:

- tool_name: Identifier for the tool
- content: Rendered context string
- priority: Priority level for context budgeting (default 70)
- learning_visible: Whether this context is visible for knowledge extraction

ToolContext is NOT stored in any memory system.

---

## Context Budget Manager

Responsibilities:

- Compile all available context into packages that fit within the active provider's context window.

Inputs:

- User Input
- Working Memory
- Semantic Memory
- Tool Context
- Chat History
- System Prompt
- Candidate Facts

Outputs:

- ReasoningContextPackage (for PromptBuilder)
- LearningContextPackage (for KnowledgeExtractor)

The Context Budget Manager:

- Detects the active provider's context window via get_context_window()
- Reserves generation budget (25% by default)
- Computes Working Memory budget from actual remaining prompt space
- Prunes Working Memory via WorkingMemory.prune() before compilation
- Orders sources by priority (100 = User Input, 60 = Chat History)
- Trims lowest-priority truncatable sources when budget is exceeded
- NEVER rewrites, summarizes, or paraphrases
- NEVER checks tool names (uses metadata instead)

---

## Reasoning Package

A compiled, budgeted set of context sources for the reasoning pipeline.

Consumed by PromptBuilder, which renders it into the final prompt.

Guaranteed to fit within the provider's context window.

---

## Learning Package

A compiled package of context sources visible for learning.

Consumed by KnowledgeExtractor.

Only sources with learning_visible=True are included.

The Context Budget Manager determines what is visible for learning.

---

## PromptBuilder

Responsibilities:

- Render a ReasoningContextPackage into a formatted prompt string.

The PromptBuilder is a pure renderer. It does NOT perform budgeting.

---

## Response Reasoner

Responsibilities:

- Generate the user response.

Inputs:

- ReasoningContextPackage (via PromptBuilder)

Outputs:

- Assistant response

---

## Knowledge Extractor

Responsibilities:

- Extract durable knowledge from a completed interaction.

Inputs:

- LearningContextPackage (conversation + optional tool context)

Outputs:

- Knowledge Candidates

The Knowledge Extractor is completely unaware of individual tools.

The Context Budget Manager determines what context is visible for learning.

Knowledge extraction never modifies Semantic Memory directly.

---

## Memory Reconciler

Responsibilities:

- Compare knowledge candidates against existing Semantic Memory to determine duplicates, conflicts, and new facts.

Inputs:

- Knowledge Candidates
- Existing Semantic Memory entries

Outputs:

- Semantic Memory updates (additions, no deletions)

The Memory Reconciler makes exactly one LLM call per candidate batch.

---

## Knowledge Validator

Responsibilities:

- Apply deterministic quality gates to reject low-quality or duplicate candidates before reconciliation.

Inputs:

- Knowledge Candidates

Outputs:

- Validated Knowledge Candidates

---

## Working Memory

Responsibilities:

- Maintain a sliding window of conversation history
- Store temporary knowledge candidates

The Working Memory:

- Evicts oldest entries via prune(max_tokens, entries) when budget is exceeded
- Is pruned twice per interaction: in-pipeline (by Context Budget Manager) and post-response (by Brain)
- Uses a deterministic token estimate: len(text) // 4
- The Context Budget Manager performs exact counting via the provider

---

## Knowledge Extractor

Responsibilities:

Extract durable knowledge from a completed interaction.

Inputs:

- LearningContextPackage (conversation + optional tool context)

Outputs:

- Knowledge Candidates

The Knowledge Extractor is completely unaware of individual tools.

The Context Budget Manager determines what context is visible for learning.

Knowledge extraction never modifies Semantic Memory directly.

---

# Athena Core Pipeline

```text
                    USER
                      │
                      ▼
                Create Thought
                      │
                      ▼
         Load Episodic Memory
                      │
                      ▼
         Retrieve Semantic Memory
                      │
                      ▼
              Tool Planner
                      │
                      ▼
              Tool Router
                      │
                      ▼
              Tool Context
                      │
                      ▼
         ┌──────────────────────┐
         │  Context Budget      │
         │  Manager             │
         │                      │
         │  Phase 1:            │
         │   Compute WM budget  │
         │   → WorkingMemory    │
         │     .prune()         │
         │                      │
         │  Phase 2:            │
         │   Compile packages   │
         └──────────┬───────────┘
                    │
         ┌──────────┴──────────┐
         │                     │
         ▼                     ▼
  Reasoning Package     Learning Package
         │                     │
         ▼                     ▼
  PromptBuilder         Knowledge Extractor
         │                     │
         ▼                     ▼
  Response Reasoner     Knowledge Candidates
      (LLM)                   │
         │                     ▼
         ▼             Knowledge Validator
  Generate Response            │
         │                     ▼
         ├────► Return   Memory Reconciler
         │      Response       │
         ▼                     ▼
  Build Completed       Semantic Memory
    Interaction              Updated
         │
         ▼
  WorkingMemory.prune()
  (post-response)
```

# Architectural Invariants

The following rules define Athena Core.

These should not be violated without changing the Core specification.

## 1.

Reasoning and Learning are independent cognitive processes.

---

## 2.

Learning never changes the response currently being generated.

---

## 3.

Semantic Memory is the only long-term factual memory used during reasoning.

---

## 4.

Knowledge extraction operates on the completed interaction.

---

## 5.

Knowledge validation is the only mechanism that may modify Semantic Memory.

---

## 6.

The Brain coordinates workers.

Workers perform cognition.

---

## 7.

Every worker has exactly one responsibility.

---

## 8.

The Core favors simplicity over optimization.

Capabilities belong in extensions whenever possible.

---

# Extension Points

Athena Core is designed to support future capabilities without architectural redesign.

Examples include:

- Vision
- Document processing
- Email and Calendar integration
- Git integration
- File management
- Scheduling
- Multi-provider orchestration
- Embeddings
- Plugins

Extensions should build upon the Core rather than replacing it.

---

# Non-Goals

Athena Core intentionally does not include:

- Autonomous agents
- Multi-agent systems
- Planning strategies
- Tool selection
- Reflection policies
- Embedding systems
- Search algorithms
- UI logic

These belong to higher-level capabilities.

---

# Core Stability Policy

Athena Core is considered architecturally frozen.

Changes to the Core should only occur if they:

- reduce architectural complexity, or
- enable an entirely new class of capability.

Behavioral improvements should generally be implemented by extending the existing architecture rather than redesigning it.
# Core Values

Athena Core is guided principles:

- Simplicity before sophistication.
- Local-first whenever practical.
- Stable foundations before new capabilities.
